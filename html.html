Stage One: The Form Element

The Form Element

The form element is like a div element, but for form controls.

Forms may not be nested inside other forms.

Forms can be processed by server side languages like ruby, php and python.

Part of processing the data involves two important form attributes; form action="the url of the program that will process information submitted via the form, and method="post or get" is the http method that the browser uses to submit the form; post corresponds to the HTTP POST method; "post" means that the html information is posted to the server, while "get" corresponds to the HTTP GET Method; in this case data is sent inside of the url, and parameters are seperated with a question mark. It's important to coordinate the client and server-side code so that the browsers can talk to one another.

The Input Element

The input element is the most commonly used form element. It is a self-closing tag, like image; no need to close it.

The input type attribute denotes what format the data will be in; several types are email, telephone, text, and so on; id refers to how we will identify the input element for css and javascript targeting; the name attribute is the variable that holds the user input data.

The Textarea Element

Sometimes a single line for text-input isn't enough; contact forms, and or user bios might require a textarea box.

The Button Element

A form is not complete without a submit button. There are also two other types of buttons; the reset button will clear all form data when clicked, and the button type "button" has no default behavior and is mostly intended for use in combination with javascript.

Stage Two: Organizing Forms

The Label Element

We can use a label element to let the user know what the following elements are meant to represent. For example:
<label for="name">Name:</label>. When the label is clicked, this gives focus to the element meant to have it.

Fieldsets and Legends

We can wrap form controls together using the fieldset element. Then we can set the child items to flex, and use css to style the group of items.

We can use a legend element just below the fieldset and just above the controls to give a group title to the form controls information context. I'll also use a span element to target just a portion of the content on the line, for styling. Example:
<legend><span class="number">1</span>Your Basic Info</legend>

Stage Three: Choosing Options

Select Menus

Anytime you have a list of more than 5 things from which you would like a user to choose, it is best to go with a select menu, because it saves space.

The elements we put inside the select element are called option elements. Check notes in this html file for an example of a select menu and option elements.

The optgroup element and its accompanying "label" attribute provides heading for the options in the dropdown menu list.

Radio Buttons

When there are fewer than 6 options from which the user may choose, it is best to go with radio buttons. The html file for this page shows an example for input radio buttons.

Since the two radio elements share the same name="user_name" attribute value, the button will toggle back and forth, allowing us to chose one or the other at any one time; if the two elements had a different name attribute value, then we could select both choices at the same time, so we need to make sure to get that correct.

Checkboxes

Checkboxes are useful for when you have multiple options to present to the user, and you want to give them the option of choosing more than one of them at once. Check the html notes for this page to see an example of checkboxes.
